# shadcn/ui

Use this skill when you need to add, update, inspect, or compose shadcn/ui components in this repository.

## When To Use

- Adding a new shadcn/ui component to the codebase.
- Updating an existing shadcn/ui component generated by the CLI.
- Debugging a component composed from `@/components/ui/*`.
- Checking how a shadcn/ui component should be used according to the official docs.

## Core Rules

- Prefer the official shadcn/ui documentation over memory.
- Keep imports local: use `@/components/ui/<component>`.
- Do not edit files under `src/components/ui/*` unless the user explicitly asks for that component change.
- Prefer composition over creating parallel UI variants.
- Preserve the repo's existing visual language and follow `AGENTS.md` plus `.context/docs/design.md`.

## Recommended Workflow

1. Check whether the component already exists locally.
2. If the component is missing, add it with the shadcn CLI.
3. Inspect the generated component before using it.
4. Compose it into the existing screen instead of rewriting shared primitives.
5. Validate with the project's typecheck or relevant tests.

## CLI Commands

- Initialize a shadcn/ui project: `pnpm dlx shadcn@latest init`
- Add a component: `pnpm dlx shadcn@latest add <component>`
- Rebuild or update an installed component only when the repo explicitly allows it.

## Guardrails

- Do not introduce a second design system.
- Do not hardcode colors or spacing that conflict with existing tokens when the repo already has them.
- Do not modify shared UI primitives just to satisfy one screen.
- If a component is being used for a new UX pattern, verify accessibility, loading states, and empty states.
